Implementazione:
- Ottenere session corrente con `await auth()` (importato da @/lib/auth)
- Se `!session?.user?.id`, throw Error('Non autenticato')
- Query credentials table per ottenere sessionToken dell'utente:
  ```typescript
  const userCreds = await db.query.credentials.findFirst({
    where: eq(credentials.userId, session.user.id),
  });
  ```
- Se `!userCreds?.sessionToken`, throw Error('Session token non trovato. Collega il tuo account PilActive dalle impostazioni.')
- Chiamare ShaggyOwl API:
  ```typescript
  const result = await shaggyOwlClient<any>({
    method: 'POST',
    endpoint: '/funzioniapp/v407/palinsesti',
    contentType: 'form',
    body: {
      codice_sessione: userCreds.sessionToken,
      language: 'it',
    },
  });
  ```
- Se `!result.success || result.data?.status !== 2`, throw Error('Impossibile recuperare gli eventi dalla palestra')
- Flatten nested structure:
  - Loop su `result.data.parametri.lista_risultati`
  - Per ogni palinsesto, loop su `palinsesto.giorni`
  - Per ogni giorno, loop su `giorno.orari_giorno`
  - Filtrare solo corsi prenotabili: `if (orario.prenotabile_corso !== '2') continue;`
  - Mappare a Event interface con parsing corretto:
    - `postiDisponibili: parseInt(orario.prenotazioni.numero_posti_disponibili)`
    - `postiOccupati: parseInt(orario.prenotazioni.numero_posti_occupati)`
    - `utentiInCoda: parseInt(orario.prenotazioni.numero_utenti_coda)`
    - `prenotabile: orario.prenotazioni.id_disponibilita === '1' && orario.prenotazioni.utente_prenotato === '0'`
    - `giaPrenotato: orario.prenotazioni.utente_prenotato === '1'`
- Applicare filtri:
  - Se `filters?.dateFrom`, filtrare eventi con `new Date(e.data) >= new Date(filters.dateFrom)`
  - Se `filters?.dateTo`, filtrare eventi con `new Date(e.data) <= new Date(filters.dateTo)`

  ```typescript
  filtered.sort((a, b) => {

    if (dateCompare !== 0) return dateCompare;
  ```

```typescript
/**
 * Formatta data relativa rispetto ad oggi
 * @example domani → "Domani", fra 5 giorni → "Tra 5 giorni"
 */
export function formatRelativeDate(date: Date | string): string {
  const dateObj = typeof date === 'string' ? new Date(date) : date;
  const now = new Date();
  
  // Reset hours to compare only dates
  const dateOnly = new Date(dateObj.getFullYear(), dateObj.getMonth(), dateObj.getDate());
  const nowOnly = new Date(now.getFullYear(), now.getMonth(), now.getDate());
  
  const diffDays = Math.ceil((dateOnly.getTime() - nowOnly.getTime()) / (1000 * 60 * 60 * 24));
  
  if (diffDays === 0) return 'Oggi';
  if (diffDays === 1) return 'Domani';
  if (diffDays === -1) return 'Ieri';
  if (diffDays > 1) return `Tra ${diffDays} giorni`;
  return `${Math.abs(diffDays)} giorni fa`;
}
```

**NOTA LOCALE**: Tutte le funzioni usano locale 'it-IT' per consistenza con l'applicazione italiana (come già stabilito in Phase 1 con `lang="it"` nel layout).

**NOTA PERFORMANCE**: Usare API native `Intl.DateTimeFormat` invece di librerie esterne (zero bundle size cost, già ottimizzate dal browser/Node.js).

- **Client → Server**: Nessun dato passa dal client in questo plan (solo server-side code)
- **Server → ShaggyOwl API**: Session token inviato per autenticazione
- **Server → Database**: Query per ottenere credentials dell'utente autenticato

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | SQL Injection | getEvents database query | mitigate | Drizzle ORM con parameterized queries (`eq(credentials.userId, session.user.id)`), no raw SQL |
| T-02-02 | Spoofing | ShaggyOwl API authentication | mitigate | Session token validated by ShaggyOwl API, stored server-side only, never exposed to client |
| T-02-03 | Information Disclosure | Event data exposure | accept | Eventi sono dati pubblici della palestra (orari corsi), no PII |
| T-02-04 | Tampering | API response modification | accept | ShaggyOwl API su HTTPS, response validation implicita nel parsing |
| T-02-05 | Denial of Service | Excessive API calls | accept | Rate limiting deferred to Phase 4, low risk in Phase 2 testing |

**Severity Assessment**: No HIGH severity threats. All threats either mitigated with ASVS L1 controls (parameterized queries, server-side auth) or accepted with documented rationale.
