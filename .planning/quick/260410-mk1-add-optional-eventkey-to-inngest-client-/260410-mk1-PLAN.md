## Background

Inngest dev server requires an eventKey for the event sending endpoint (`/e/<EVENT_KEY>`). From Inngest documentation:
- Production: eventKey authenticates requests to Inngest Platform
- Local dev: Can use dummy value since dev server doesn't validate keys locally
- Events are sent to: `/e/<EVENT_KEY>` endpoint

Current client configuration (lib/inngest/client.ts) only has `id` and `env` fields — missing optional `eventKey` field needed for dev server endpoint routing.

This addresses Open Question #2 from Phase 03 RESEARCH.md about Inngest local development workflow.
</context>

**Update lib/inngest/client.ts:**

Add `eventKey` field to Inngest client configuration, reading from environment variable:

```typescript
import { Inngest } from "inngest";

export const inngest = new Inngest({
  id: "pilactive-helper",
  env: process.env.INNGEST_ENV ?? "development",
  eventKey: process.env.INNGEST_EVENT_KEY,
});
```

**Update .env.local:**

Add INNGEST_EVENT_KEY with dummy value for local development (create .env.local if it doesn't exist):

```
INNGEST_EVENT_KEY=local-dev-key
```

**Why "local-dev-key" as dummy value:**
- Dev server doesn't validate eventKey locally (per Inngest docs: "you can use a dummy value")
- Value is used for endpoint routing (`/e/<EVENT_KEY>`) but not authentication in local dev
- For production deployment, replace with real event key from Inngest dashboard

**Pattern rationale:**
- Optional field (`eventKey` can be undefined in theory, but needed for dev server)
- Environment-based configuration (follows existing pattern for `INNGEST_ENV`)
- Matches Inngest documentation for local development setup
