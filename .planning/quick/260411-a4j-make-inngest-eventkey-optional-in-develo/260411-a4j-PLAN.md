## Problem Context

User is experiencing 404 errors ("Branch environment does not exist") when running Inngest dev server locally with production signing keys. The issue is that eventKey is designed for cloud event routing to branch environments - it's not needed for local development.

## Current State

```typescript
export const inngest = new Inngest({
  id: "pilactive-helper",
  env: process.env.INNGEST_ENV ?? "development",
  eventKey: process.env.INNGEST_EVENT_KEY,
});
```

This configuration always includes eventKey, even in development. When INNGEST_EVENT_KEY contains a production signing key, the local dev server tries to route to a cloud branch that doesn't exist.

## Solution Pattern

From Inngest documentation and best practices:
- eventKey is for cloud event routing (production/staging branches)
- Local dev server doesn't need eventKey - it routes directly
- Use environment-based conditional configuration
- Only include eventKey when INNGEST_ENV is production (or similar cloud environment)

## Stack Context

- Inngest 3.0+ with branch environments support
- Next.js 16.2+ environment variable patterns
- TypeScript strict mode enabled

Change from:
```typescript
export const inngest = new Inngest({
  id: "pilactive-helper",
  env: process.env.INNGEST_ENV ?? "development",
  eventKey: process.env.INNGEST_EVENT_KEY,
});
```

To:
```typescript
export const inngest = new Inngest({
  id: "pilactive-helper",
  env: process.env.INNGEST_ENV ?? "development",
  ...(process.env.INNGEST_ENV === "production" && {
    eventKey: process.env.INNGEST_EVENT_KEY,
  }),
});
```

**Why this pattern:**
- Uses object spread with conditional inclusion (TypeScript best practice)
- Only includes eventKey property when INNGEST_ENV is exactly "production"
- In development (default), eventKey is omitted entirely from config object
- In production (Vercel deployment), eventKey is included from environment variable
- No undefined values passed (cleaner than `eventKey: condition ? value : undefined`)

**Alternative considered:** Could use `process.env.INNGEST_ENV !== "development"` but explicit production check is safer (prevents accidental inclusion in staging if env is misconfigured).
